# Quick Task: Increase scan throughput — all players every 5 min

## Problem
With 1,772 players, the scanner completes a full sweep in ~88 minutes instead of the
intended 5 minutes. Two bottlenecks:

1. `dispatch_scans()` picks up only `SCAN_CONCURRENCY * 2 = 10` players per dispatch cycle
2. `SCAN_CONCURRENCY = 5` limits parallel API calls to 5

Math: 10 players per 30s dispatch = 20/min = 1,200/hr → 88 min per sweep.

## Target
1,772 players / 5 min = 354 players/min ≈ 6 players/sec.
Each scan = 2 API calls × ~0.5s. Need ~30-40 concurrent workers.

## Plan

### T1: Raise SCAN_CONCURRENCY in config.py
- Change `SCAN_CONCURRENCY = 5` → `SCAN_CONCURRENCY = 40`
- This controls the semaphore in `dispatch_scans()`

### T2: Remove batch limit in dispatch_scans()
- Remove `.limit(SCAN_CONCURRENCY * 2)` from the query — pick up ALL due players
- The semaphore already gates actual concurrency, so the limit just starves the queue

### T3: Reduce per-request sleep in futgg_client._get()
- Reduce `asyncio.sleep(0.15)` → `asyncio.sleep(0.05)`
- With 40 concurrent workers, total API rate stays under 800 req/min
- fut.gg is a public API with no visible rate limiting headers

## Risk
- fut.gg could rate-limit at higher concurrency. Mitigated by circuit breaker +
  retry logic already in place. Can dial back SCAN_CONCURRENCY if needed.
