# Phase 09.1: Migrate from SQLite - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the backend from SQLite (aiosqlite) to PostgreSQL (asyncpg). This includes: swapping the database engine, migrating 7GB of production data, removing SQLite-specific workarounds (3-engine split, volatility cache), cleaning up schema remnants, and fixing the server bugs that prevented Phase 9 integration tests from passing. The migration is complete when all Phase 9 tests pass against Postgres.

</domain>

<decisions>
## Implementation Decisions

### Hosting & Connection
- **D-01:** PostgreSQL runs in Docker via docker-compose. Local-first, zero cost, easy to recreate. Cloud migration later without code changes.
- **D-02:** Connection string managed via `DATABASE_URL` environment variable. `.env` file for local dev, docker-compose sets it automatically. Current `config.py` env-var pattern preserved — just change the fallback default.
- **D-03:** Consolidate the 3 SQLite engines (scanner write, API read, API write) into a single Postgres engine with asyncpg connection pool. Postgres MVCC eliminates the need for separate engines.

### Data Migration
- **D-04:** Use pgloader for one-shot SQLite → Postgres data migration. Handles type mapping, indexes, constraints. ~5-10 min for 7GB.
- **D-05:** Brief downtime is acceptable. Stop scanner, run pgloader, restart with Postgres URL.
- **D-06:** Keep SQLite file as backup after migration. Don't delete the .db file — enables quick rollback by reverting DATABASE_URL.

### Dialect Swap
- **D-07:** Rewrite `health_check.py` from raw `sqlite3` to async SQLAlchemy, matching the rest of the codebase. Eliminates the last raw SQLite dependency.
- **D-08:** Replace `sqlite_insert` upserts (4 locations in listing_tracker.py and scanner.py) with PostgreSQL-compatible `insert(...).on_conflict_do_update()`.
- **D-09:** Remove all SQLite PRAGMAs (WAL, synchronous, query_only, foreign_keys) from db.py. Postgres handles these natively.
- **D-10:** Clean up schema remnants during migration: drop `futbin_id` column, set sensible `scan_tier` defaults, backfill `scorer_version` NULLs. Fresh start on Postgres with a clean schema.

### Test & Verification
- **D-11:** Integration tests use a Docker test container (testcontainers-python or docker-compose). Each test run gets a fresh Postgres DB. No state leaks.
- **D-12:** Verification bar: all Phase 9 integration tests must pass on Postgres. The Phase 9 tests currently fail due to SQLite server-side issues — this migration is part of fixing the server, not just a DB swap.
- **D-13:** Remove `volatility_cache.py` workaround. Postgres with proper indexing makes the volatility query fast enough (<1s). Simpler code, fewer layers.

### Claude's Discretion
- pgloader configuration details (type mappings, index handling)
- asyncpg connection pool sizing
- docker-compose configuration specifics
- Post-migration SQL cleanup script details
- Whether to use Alembic for schema management or keep `create_all` approach

</decisions>
