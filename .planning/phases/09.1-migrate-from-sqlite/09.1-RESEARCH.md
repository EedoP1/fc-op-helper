## Summary

This phase swaps the SQLite/aiosqlite backend for PostgreSQL/asyncpg, fixing write-lock contention that caused the Phase 9 integration tests to fail. The existing codebase is nearly dialect-agnostic — SQLAlchemy ORM models are unchanged, and the only SQLite-specific code is: 4 `sqlite_insert` upserts, 3 PRAGMA event listeners, the 3-engine split in `db.py`, and the raw `sqlite3` module in `health_check.py`.

The migration path is straightforward: install `asyncpg`, swap the engine connection string, consolidate 3 engines into 1, replace dialect-specific inserts with the PostgreSQL-compatible dialect, and stand up a Docker Compose PostgreSQL container. Data migration via pgloader is a one-shot operation. The test harness needs to swap its `aiosqlite` teardown for a Postgres-backed testcontainer.

**Primary recommendation:** Use `sqlalchemy.dialects.postgresql.insert` as a drop-in replacement for `sqlalchemy.dialects.sqlite.insert`. The `on_conflict_do_update()` API is identical. The asyncpg engine URL is `postgresql+asyncpg://user:pass@host:port/db`. Keep `Base.metadata.create_all` — no Alembic needed for this phase.

### Pattern 1: Single Async Engine with asyncpg

**What:** Replace the 3-engine SQLite split with a single `create_async_engine` using `postgresql+asyncpg://`. Postgres MVCC allows concurrent readers/writers on a single connection pool.

**When to use:** Always — the SQLite 3-engine split was a workaround for SQLite serialized writes.

```python
# src/server/db.py — new create_engine()
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession, AsyncEngine
from src.config import DATABASE_URL

def create_engine(db_url: str = DATABASE_URL) -> AsyncEngine:
    """Create a single async engine backed by asyncpg connection pool."""
    return create_async_engine(
        db_url,
        echo=False,
        pool_size=10,
        max_overflow=20,
        pool_pre_ping=True,  # validates connections before checkout
    )
```

**Pool sizing guidance (Claude's discretion):**
- `pool_size=10` handles scanner concurrency (40 concurrent scans) because asyncpg connections are lightweight and the pool is shared across async tasks
- `max_overflow=20` permits burst headroom during bootstrap
- `pool_pre_ping=True` is required for long-running servers (Docker restarts)

### Pattern 2: PostgreSQL Upsert (Drop-in Replacement)

**What:** Replace `from sqlalchemy.dialects.sqlite import insert as sqlite_insert` with `from sqlalchemy.dialects.postgresql import insert as pg_insert`. The `on_conflict_do_update()` API is syntactically identical.

**4 locations to change:**

| File | Line | Table | Conflict Target |
|------|------|-------|----------------|
| `scanner.py:125` | `PlayerRecord` upsert (bootstrap) | `["ea_id"]` |
| `scanner.py:219` | `PlayerRecord` upsert (discovery) | `["ea_id"]` |
| `listing_tracker.py:153` | `ListingObservation` upsert | `["fingerprint"]` |
| `listing_tracker.py:334` | `DailyListingSummary` upsert | `["id"]` (**BUG — see Pitfall 4**) |

```python

stmt = sqlite_insert(PlayerRecord).values(**row)
stmt = stmt.on_conflict_do_update(
    index_elements=["ea_id"],
    set_=dict(name=row["name"], ...),

# After (PostgreSQL dialect — identical API)

stmt = pg_insert(PlayerRecord).values(**row)
stmt = stmt.on_conflict_do_update(
    index_elements=["ea_id"],
    set_=dict(name=row["name"], ...),
)
```

async def _select_players(session: AsyncSession, count: int) -> list[dict]:
    """Select players via SQLAlchemy ORM instead of raw sqlite3."""
    stmt = (
        select(PlayerRecord)
        .where(PlayerRecord.is_active == True)

### Anti-Patterns to Avoid

- **Keeping 3 engines:** The 3-engine split was a SQLite workaround. On Postgres, it adds connection overhead with no benefit. One pool is correct.
- **SQLite connect_args on asyncpg engine:** `connect_args={"timeout": 30}` is SQLite-specific. asyncpg uses `connect_args={"command_timeout": 30}` with different semantics — remove from scanner engine, adjust if needed for API write timeouts.
- **WAL / synchronous PRAGMAs:** These are SQLite-only. The `event.listens_for(engine.sync_engine, "connect")` blocks in db.py must be removed entirely.
- **aiosqlite in conftest cleanup:** The `cleanup_tables` fixture in conftest.py uses `create_async_engine("sqlite+aiosqlite://...")`. This must be replaced with the Postgres cleanup approach.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Postgres upsert on conflict | Custom SELECT+INSERT logic | `pg_insert().on_conflict_do_update()` | SQLAlchemy handles locking, atomicity, and the conflict detection correctly |
| Ephemeral test DB | Manual Docker subprocess with port management | `testcontainers[postgres]` | Handles container lifecycle, port allocation, readiness polling, and cleanup automatically |
| Data migration from SQLite | Custom Python script reading SQLite, writing Postgres | `pgloader` | Handles type coercion (SQLite TEXT→TIMESTAMP), index creation, constraint mapping, and is proven for GB-scale migrations |
| Connection pool management | Manual connection tracking | asyncpg pool via SQLAlchemy engine | `pool_pre_ping=True` + pool_size handles all reconnection and health-check logic |

**Key insight:** The SQLite dialect inserts (`sqlite_insert`) and the PostgreSQL dialect inserts (`pg_insert`) have identical Python APIs — the only code change is the import line and the import alias. This is a near-zero-risk swap.

## Common Pitfalls

### Pitfall 1: asyncpg datetime naive/aware mismatch
**What goes wrong:** asyncpg returns timezone-aware `datetime` objects (UTC). SQLAlchemy ORM columns declared as `DateTime` (no timezone) store naive datetimes. Mismatches cause comparison failures or `TypeError` in application code that does arithmetic between naive and aware datetimes.
**Why it happens:** SQLite `aiosqlite` returns naive strings; asyncpg returns aware objects. The application's `datetime.utcnow()` calls produce naive datetimes — these will mismatch the aware datetimes returned from Postgres queries.
**How to avoid:** Either (a) use `DateTime(timezone=True)` on all ORM columns and `datetime.now(timezone.utc)` in application code, or (b) configure asyncpg to return naive datetimes via `connect_args={"server_settings": {"TimeZone": "UTC"}}` and strip tzinfo at the ORM level. **Option (a) is cleaner but requires updating all models and all `datetime.utcnow()` calls in scanner.py and listing_tracker.py.** Option (b) can be done in db.py without touching models.
**Warning signs:** `TypeError: can't compare offset-naive and offset-aware datetimes` in logs; test failures on any query that filters by datetime column.

### Pitfall 2: pgloader type mapping for SQLite text-encoded datetimes
**What goes wrong:** SQLite stores datetimes as TEXT (`"2026-03-27 22:24:24.599680"`). pgloader must map these to `TIMESTAMP WITHOUT TIME ZONE` in Postgres. If pgloader uses the default type mapping, some columns may land as TEXT in Postgres.
**Why it happens:** SQLite has no native DATETIME type; it stores as TEXT or INTEGER. pgloader's default affinity rules handle most cases, but edge cases (NULL datetimes, ISO 8601 with microseconds) can fail.
**How to avoid:** Test pgloader with a subset of rows first. Specify explicit CAST rules in the pgloader `.load` file for datetime columns:
```
CAST type datetime to timestamptz using zero-dates-to-null
```
**Warning signs:** `invalid input syntax for type timestamp` errors in Postgres after migration; datetime columns showing up as text type in `\d+ players`.

### Pitfall 3: `pool_pre_ping=True` and asyncpg
**What goes wrong:** `pool_pre_ping=True` on asyncpg engines may issue a `SELECT 1` on every connection checkout if the pool was idle. For a scanner dispatching 40 concurrent tasks, this adds latency.
**Why it happens:** asyncpg connections go stale after idle periods (Docker restarts, network blips). `pool_pre_ping` re-validates them.
**How to avoid:** Keep `pool_pre_ping=True` for correctness; the overhead is negligible compared to network RTT. Do not disable it to "optimize" — connection errors without pre_ping are far worse.
**Warning signs:** N/A — this pitfall is about incorrect optimization attempts, not a failure mode.

### Pitfall 4: DailyListingSummary upsert uses wrong conflict target (pre-existing bug)
**What goes wrong:** `listing_tracker.py:334` uses `on_conflict_do_update(index_elements=["id"])`. Since `id` is an autoincrement primary key, it never conflicts — the upsert always inserts a new row instead of updating. This means `daily_listing_summaries` accumulates duplicate `(ea_id, date, margin_pct)` rows.
**Why it happens:** This is a pre-existing bug hidden by the fact that `daily_listing_summaries` is always empty (resolution pipeline is broken — per db-integrity-audit.md). On Postgres, this bug becomes visible as duplicates.
**How to avoid:** Fix the conflict target during migration to `["ea_id", "date", "margin_pct"]`. The `DailyListingSummary` table has a composite index `ix_daily_summary_ea_id_date_margin` on those columns — add a `UniqueConstraint` to make it a proper conflict target:
```python
# models_db.py — add to DailyListingSummary.__table_args__
UniqueConstraint("ea_id", "date", "margin_pct", name="uq_daily_summary_ea_id_date_margin"),
```
**Warning signs:** `daily_listing_summaries` row count growing as a multiple of `len(MARGINS)` per `aggregate_daily_summaries()` call; duplicate rows visible with `SELECT ea_id, date, margin_pct, COUNT(*) FROM daily_listing_summaries GROUP BY 1,2,3 HAVING COUNT(*) > 1`.

### Pitfall 5: Server harness still references 3 engines after consolidation
**What goes wrong:** `tests/integration/server_harness.py` explicitly creates and wires 3 engines (`create_engine_and_tables`, `create_read_engine`, `create_api_write_engine`). After db.py is rewritten to 1 engine, the harness must be updated to match.
**Why it happens:** The harness mirrors `src/server/main.py` lifespan but also sets `app.state.scanner_session_factory` separately (line 106). After consolidation, all session factories point to the same engine.
**How to avoid:** Update server_harness.py and src/server/main.py lifespan together in the same task. Verify `app.state.session_factory`, `app.state.read_session_factory`, and `app.state.scanner_session_factory` all point to the single engine's session factory.
**Warning signs:** `AttributeError: app.state has no attribute read_session_factory` in API endpoints; or engines disposed in the wrong order on shutdown.

### Pitfall 6: testcontainers on Windows requires Docker Desktop running
**What goes wrong:** `testcontainers.postgres.PostgresContainer` requires the Docker daemon to be accessible. On Windows, this means Docker Desktop must be running. If Docker Desktop is not running when tests start, testcontainers raises `DockerException`.
**Why it happens:** testcontainers uses the Docker SDK (`docker` Python package) which connects to the Docker socket. On Windows, the socket path is different from Linux.
**How to avoid:** Install Docker Desktop. Start Docker Desktop before running integration tests. Add a pre-flight check in conftest.py that raises a clear error if Docker is unavailable:
```python
import docker
def _check_docker():
    try:
        docker.from_env().ping()
    except Exception:
        pytest.skip("Docker not available — skipping integration tests")
```
**Warning signs:** `docker.errors.DockerException: Error while fetching server API version` at test collection time.

### Pitfall 7: pgloader not available on Windows natively
**What goes wrong:** pgloader is a Linux/Mac binary. It is not available via pip or chocolatey. On Windows, pgloader must be run via Docker or WSL.
**Why it happens:** pgloader is written in Common Lisp and has no Windows build.
**How to avoid:** Use pgloader via Docker: `docker run --rm --network host dimitri/pgloader pgloader <options>`. This requires the SQLite file to be volume-mounted into the container and the Postgres container to be accessible from the pgloader container. Plan the migration step as a Docker command, not a direct CLI call.
**Warning signs:** `pgloader not found` when trying to run natively on Windows.

### SQLAlchemy asyncpg Engine Creation (Production)

```python
# src/server/db.py — after migration
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession, AsyncEngine
from sqlalchemy.orm import DeclarativeBase
from src.config import DATABASE_URL


class Base(DeclarativeBase):
    pass


def create_engine(db_url: str = DATABASE_URL) -> AsyncEngine:
    """Create a single async Postgres engine with asyncpg connection pool."""
    return create_async_engine(
        db_url,
        echo=False,
        pool_size=10,
        max_overflow=20,
        pool_pre_ping=True,
    )


def create_session_factory(engine: AsyncEngine) -> async_sessionmaker[AsyncSession]:
    """Create an async session factory bound to the given engine."""
    return async_sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)


async def create_engine_and_tables(
    db_url: str = DATABASE_URL,
) -> tuple[AsyncEngine, async_sessionmaker[AsyncSession]]:
    """Create Postgres engine, create all ORM tables, return engine and session factory."""
    engine = create_engine(db_url)
    async with engine.begin() as conn:
        from src.server.models_db import (  # noqa: F401
            PlayerRecord, PlayerScore, MarketSnapshot, SnapshotSale,
            SnapshotPricePoint, ListingObservation, DailyListingSummary,
            TradeAction, TradeRecord, PortfolioSlot,
        )
        await conn.run_sync(Base.metadata.create_all)
    return engine, create_session_factory(engine)
```

   - Recommendation: Use `connect_args={"server_settings": {"TimeZone": "UTC"}}` in the asyncpg engine to suppress timezone info on returned datetimes. This avoids touching 20+ `datetime.utcnow()` call sites. If tests reveal comparison failures, escalate to full aware-datetime refactor.
