# Phase 07.1: Trade Reporting - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Extension passively reads the EA Web App DOM (transfer list, trade pile) to detect trade outcomes (bought, listed, sold, expired) for portfolio players and auto-reports them to the backend. Observation only — no automated clicking, no buy/list/relist actions. Includes DOM exploration task to map EA Web App transfer list selectors. This phase provides the data pipeline that Phase 07.2's dashboard consumes.

</domain>

<decisions>
## Implementation Decisions

### DOM Observation Scope
- **D-01:** Extension reads the EA Web App transfer list page when the user navigates to it. Passive scan — only reads DOM when user is already on that page, no automated navigation.
- **D-02:** Track all trade outcomes: bought (player appears in trade pile after purchase), listed (player appears in active transfer list), sold (completed sale in transfer list), expired (expired listing in transfer list).
- **D-03:** Only track players that are in the confirmed portfolio (match by ea_id against portfolio_slots). Ignore non-portfolio player activity.

### DOM Exploration
- **D-04:** Phase includes a DOM exploration task to inspect the EA Web App transfer list structure via live DevTools. Document selectors for player items, prices, statuses, and player identification (ea_id or name matching).
- **D-05:** Selectors centralized in a single file (aligns with Phase 8 requirement AUTO-08). Phase 8 extends this selector map for automation actions.

### Reporting Behavior
- **D-06:** Auto-report silently. When the extension detects a trade outcome for a portfolio player, it immediately reports to the backend via `POST /api/v1/actions/{id}/complete`. No user confirmation step.
- **D-07:** Deduplication — extension tracks which outcomes have already been reported (via chrome.storage.local or in-memory set) to avoid double-reporting on re-renders or page refreshes.

### Integration with Existing Action Queue
- **D-08:** Trade reporting reuses the existing action queue infrastructure from Phase 5. The `_derive_next_action` function already handles the continuous cycle (sold → new BUY). Reporting outcomes feeds data into this existing system.
- **D-09:** For the initial state (portfolio confirmed but no actions exist yet), the extension's first scan of the transfer list may need to bootstrap — detect current state and report it so the action queue knows where each player is in the cycle.

### Claude's Discretion
- Exact DOM selectors (discovered during exploration task)
- How to match DOM player elements to portfolio ea_ids (name matching, data attributes, or EA internal IDs)
- Content script architecture for the DOM reader (MutationObserver vs periodic scan when on transfer list page)
- Error handling for DOM structure changes or missing elements
- Storage format for deduplication tracking
- Whether to extend the existing content script or add a separate module for trade observation

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ea-webapp.content.ts` — Content script already injected on EA Web App pages with MutationObserver for SPA navigation. Trade observer logic plugs into this.
- `messages.ts` — Discriminated union pattern to extend with TRADE_REPORT message type.
- `POST /actions/{id}/complete` endpoint — Already accepts outcome reports. Trade observer calls this via service worker proxy.
- `_derive_next_action()` in actions.py — Already handles the sold → new BUY cycle. Reporting outcomes keeps this system fed.

### Established Patterns
- Content script uses MutationObserver to detect SPA navigation changes
- All backend calls route through service worker (content script never calls backend directly)
- Discriminated union messages for type-safe communication
- chrome.storage.local for state that survives service worker termination

### Integration Points
- Content script detects when user is on transfer list page (URL or DOM marker)
- Content script reads DOM for player items, statuses, prices
- Content script sends TRADE_REPORT messages to service worker
- Service worker proxies to POST /actions/{id}/complete
- Selector map file shared between this phase and Phase 8

</code_context>
