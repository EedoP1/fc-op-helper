# Phase 07.1 Plan 02: Direct Trade Record Endpoint Summary

Bootstrap trade reporting endpoint that allows the extension to record trade outcomes without a pre-existing TradeAction — solving Pitfall 2 from the trade reporting research.

## What Was Built

`POST /api/v1/trade-records/direct` accepts `{ea_id, price, outcome}` and inserts a TradeRecord row directly, bypassing the action queue. This unblocks the extension's first scan of the Transfer List after portfolio confirmation, when no TradeActions exist yet.

## Key Implementation Details

**`DirectTradeRecordPayload`** — Pydantic model added to `actions.py`:
- Fields: `ea_id: int`, `price: int`, `outcome: str`

**`_OUTCOME_TO_ACTION_TYPE`** — Static mapping dict:
- `"bought"` → `"buy"`
- `"listed"` → `"list"`
- `"sold"` → `"list"` (sold is the result of a list action)
- `"expired"` → `"list"` (expired is the result of a list action)

**Endpoint behavior:**
1. Look up `outcome` in `_OUTCOME_TO_ACTION_TYPE` — return 400 if not found
2. Query `portfolio_slots` for `ea_id` — return 404 if missing
3. Insert `TradeRecord` with derived `action_type`
4. Return 201 with `{"status": "ok", "trade_record_id": <int>}`
