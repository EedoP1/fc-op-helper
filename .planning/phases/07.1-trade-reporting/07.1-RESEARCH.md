## User Constraints (from CONTEXT.md)

### Locked Decisions

**DOM Observation Scope**
- D-01: Extension reads the EA Web App transfer list page when the user navigates to it. Passive scan — only reads DOM when user is already on that page, no automated navigation.
- D-02: Track all trade outcomes: bought (player appears in trade pile after purchase), listed (player appears in active transfer list), sold (completed sale in transfer list), expired (expired listing in transfer list).
- D-03: Only track players that are in the confirmed portfolio (match by ea_id against portfolio_slots). Ignore non-portfolio player activity.

**DOM Exploration**
- D-04: Phase includes a DOM exploration task to inspect the EA Web App transfer list structure via live DevTools. Document selectors for player items, prices, statuses, and player identification (ea_id or name matching).
- D-05: Selectors centralized in a single file (aligns with Phase 8 requirement AUTO-08). Phase 8 extends this selector map for automation actions.

**Reporting Behavior**
- D-06: Auto-report silently. When the extension detects a trade outcome for a portfolio player, it immediately reports to the backend via `POST /api/v1/actions/{id}/complete`. No user confirmation step.
- D-07: Deduplication — extension tracks which outcomes have already been reported (via chrome.storage.local or in-memory set) to avoid double-reporting on re-renders or page refreshes.

**Integration with Existing Action Queue**
- D-08: Trade reporting reuses the existing action queue infrastructure from Phase 5. The `_derive_next_action` function already handles the continuous cycle (sold → new BUY). Reporting outcomes feeds data into this existing system.
- D-09: For the initial state (portfolio confirmed but no actions exist yet), the extension's first scan of the transfer list may need to bootstrap — detect current state and report it so the action queue knows where each player is in the cycle.

## Summary

Phase 07.1 adds passive trade outcome detection to the existing Chrome extension. When the user navigates to the EA Web App Transfer List page, the content script reads the DOM to detect trade states (bought/listed/sold/expired) for portfolio players and silently reports them to the backend via the existing `POST /api/v1/actions/{id}/complete` endpoint.

The biggest uncertainty is the EA Web App DOM structure — EA uses a SPA with custom elements and obfuscated class names that change with game updates. No public documentation exists. The DOM exploration task (D-04) is a hard prerequisite: no reader code can be written until selectors are discovered in a live session. The planner must front-load exploration as Wave 0 or the first task of Wave 1.

The existing extension infrastructure is well-suited for this work. The content script already has a MutationObserver for SPA navigation, a discriminated union message bus, chrome.storage.local for persistent state, and a service worker proxy for all backend calls. This phase adds: (1) a selector map file, (2) a transfer list reader module, (3) new TRADE_REPORT message types, (4) deduplication state in storage, and (5) a service worker handler that calls `POST /actions/{id}/complete`.

**Primary recommendation:** Structure the phase as two sequential waves — Wave 0 is DOM exploration (manual DevTools task producing a filled-in `selectors.ts`), Wave 1 is the implementation that consumes those selectors. This matches how Phase 8 was planned and respects the LOW-confidence DOM knowledge.

### Pattern 2: Transfer List Reader Module

**What:** A pure function that reads the current DOM and returns a list of detected player items with their status. Stateless — no side effects. Easily unit-testable with jsdom.

**When to use:** Called once on initial page load and again on DOM mutations (new items appear).

**Example:**
```typescript
// extension/src/trade-observer.ts

  playerName: string;
  status: 'listed' | 'sold' | 'expired' | 'bought';
  price: number;
};

/**
 * Read the current transfer list DOM and return detected items.
 * Returns empty array if selectors do not match (DOM changed or wrong page).
 */
export function readTransferList(root: Document | Element = document): DetectedItem[] {
  const items = root.querySelectorAll(TRANSFER_LIST_ITEM);
  // ... parse each item
}
```

### Pattern 3: Deduplication via chrome.storage.local

**What:** A `Set<string>` serialised to `chrome.storage.local` using the existing WXT typed storage pattern. Keys are `{ea_id}:{outcome}:{price}` or similar composite strings to uniquely identify a reported outcome.

**When to use:** Before every report call — check if already reported. After a successful report — add to the set.

**Example:**
```typescript
// extension/src/storage.ts (extend existing file)
export const reportedOutcomesItem = storage.defineItem<string[]>(
  'local:reportedOutcomes',
  { fallback: [] },
);
```

### Pattern 4: Message Types for Trade Reporting

**What:** Extend the discriminated union in `messages.ts` with `TRADE_REPORT` (content script → service worker) and `TRADE_REPORT_RESULT` (service worker → content script).

**Example:**
```typescript
// extension/src/messages.ts (extend existing union)
| { type: 'TRADE_REPORT'; action_id: number; price: number; outcome: 'bought' | 'listed' | 'sold' | 'expired' }
| { type: 'TRADE_REPORT_RESULT'; success: boolean; error?: string }
```

### Pattern 5: Content Script Integration

**What:** Wire the trade observer into the existing content script lifecycle. Page detection (is the user on the Transfer List?) gates the observer. MutationObserver detects new player items appearing in the transfer list after page transitions.

**When to use:** Observer activates only when the transfer list container is present in the DOM.

**Example approach:**
```typescript
// In ea-webapp.content.ts, after SPA navigation detection:
function maybeStartTradeObserver() {
  const transferListContainer = document.querySelector(TRANSFER_LIST_CONTAINER);
  if (!transferListContainer) return; // not on transfer list page
  // start MutationObserver on the container
  // do initial scan
}
```

### Pattern 6: Bootstrap Scan (D-09)

**What:** On first content script load (or after confirm), scan the transfer list immediately to report current state so the action queue knows where each player is. Without this, the queue would generate a BUY action for a player that is already listed.

**Key logic:**
- If a portfolio player's card shows "Listed" → report outcome `listed` (so `_derive_next_action` skips it)
- If a portfolio player's card shows "Sold" → report outcome `sold` (so queue restarts BUY)
- If a portfolio player's card shows "Expired" → report outcome `expired` (so queue generates RELIST)

**Important:** To call `POST /actions/{id}/complete`, the content script needs an `action_id`. If no matching IN_PROGRESS or PENDING action exists for the player, the backend endpoint 404s. The planner must address the bootstrap case: either create a lightweight `POST /api/v1/trade-records/direct` endpoint for reporting without an action_id, or ensure the backend always has an action queued before a scan is needed.

### Anti-Patterns to Avoid

- **Calling backend directly from content script:** All backend calls must route through the service worker (Chrome CORS + MV3 architecture decision from Phase 6). Content script sends TRADE_REPORT message; service worker calls the backend.
- **Hardcoding selectors inline:** All selectors live in `selectors.ts` only (AUTO-08).
- **Scanning pages other than the transfer list:** Observer only activates when the transfer list container is present — no scanning on other EA Web App pages.
- **Re-reporting on DOM refresh without deduplication:** EA SPA often re-renders the same items. Always check `reportedOutcomesItem` before reporting.
- **Blocking content script startup on observer:** Trade observer activation is separate from content script init. Observer starts lazily when the right page is detected.

## Common Pitfalls

### Pitfall 1: DOM Exploration Must Precede Code

**What goes wrong:** Reader code is written with guessed selectors that do not match EA's actual DOM. EA uses minified/obfuscated class names that change every game update.

**Why it happens:** The only way to know the real selectors is to inspect the live EA Web App in DevTools while logged in.

**How to avoid:** DOM exploration is Wave 0 or the first task of Wave 1. All subsequent tasks block on the filled-in `selectors.ts`. The planning must enforce this dependency explicitly.

**Warning signs:** Selectors that look like `.player-card` or `#transferList` — these are almost certainly wrong for EA's SPA.

### Pitfall 2: Bootstrap Action ID Problem

**What goes wrong:** `POST /actions/{id}/complete` requires a valid `action_id`. If no action exists for a player (common on first run after portfolio confirm), the backend returns 404 and the outcome is lost.

**Why it happens:** `_derive_next_action` creates actions lazily when `GET /actions/pending` is called. If the extension reports a trade before polling has created an action, there is no action_id to reference.

**How to avoid:** Research and plan the bootstrap path explicitly. Options:
1. Add a `POST /api/v1/trade-records/direct` endpoint that accepts `{ea_id, outcome, price}` without requiring an action_id — inserts a TradeRecord directly.
2. On bootstrap scan, have the extension call `GET /actions/pending` first (which creates the action), then use the returned action_id.

Option 2 is simpler and reuses existing infrastructure but may race: the derived action type (BUY) may not match the observed state (already listed).

**Recommendation:** Add `POST /api/v1/trade-records/direct` for direct outcome injection. The planner should include a backend task for this endpoint.

### Pitfall 3: Double-Reporting on SPA Re-renders

**What goes wrong:** EA's SPA re-renders the transfer list DOM on navigation or data refresh. MutationObserver fires again; observer re-reports already-reported outcomes.

**Why it happens:** The DOM changes are indistinguishable from new items appearing.

**How to avoid:** Deduplication key stored in `reportedOutcomesItem`. Key should be composite: `{ea_id}:{outcome}:{price}` or `{ea_id}:{outcome}:{timestamp_approx}`. Check before every report call, write immediately after success.

**Warning signs:** The same trade record appears multiple times in `trade_records` for the same player within seconds.

### Pitfall 4: EA Web App Pages vs Transfer List Page

**What goes wrong:** Content script is injected on all EA Web App pages (broad match). Trade observer fires on wrong pages (home, market search, squad builder) and either finds no items or finds unrelated DOM elements.

**Why it happens:** The content script match pattern is `https://www.ea.com/ea-sports-fc/ultimate-team/web-app/*` — all pages.

**How to avoid:** Gate the trade observer on a page-presence check. Either: (a) check the URL path for transfer-list indicators, or (b) check for the transfer list container element in the DOM. Option (b) is more robust because EA may use hash/query routing that doesn't change the path.

### Pitfall 5: assertNever Compile Error When Adding Message Types

**What goes wrong:** Adding `TRADE_REPORT` and `TRADE_REPORT_RESULT` to the `ExtensionMessage` union without handling them in the content script switch causes a TypeScript compile error at `assertNever(msg)`.

**Why it happens:** `assertNever` is intentional — it enforces exhaustive handling of all message variants.

**How to avoid:** After extending `messages.ts`, add explicit cases to the content script switch (`TRADE_REPORT_RESULT` is a response type — content script handles it with `return false` like `PORTFOLIO_LOAD_RESULT`; `TRADE_REPORT` is a request sent TO the service worker — also `return false`).

### Pitfall 6: MutationObserver Scope — Observing Too High

**What goes wrong:** Observing `document.body` with `subtree: true` fires for every DOM change on the entire page (animations, tooltips, etc.), causing the observer to run far too often.

**Why it happens:** EA's SPA makes frequent small DOM updates.

**How to avoid:** Scope the observer to the transfer list container element only, with `childList: true` and `subtree: false` (or minimal subtree depth). Only re-scan when direct children of the container change (player cards being added/removed).

### Service worker handler for trade report

```typescript
// Source: extension/entrypoints/background.ts (existing proxy pattern)
case 'TRADE_REPORT':
  handleTradeReport(msg.action_id, msg.price, msg.outcome).then(sendResponse);
  return true; // async response

async function handleTradeReport(
  action_id: number,
  price: number,
  outcome: string,
): Promise<ExtensionMessage> {
  try {
    const res = await fetch(`${BACKEND_URL}/api/v1/actions/${action_id}/complete`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ price, outcome }),
    });
    if (!res.ok) {
      return { type: 'TRADE_REPORT_RESULT', success: false, error: `Backend error: ${res.status}` };
    }
    return { type: 'TRADE_REPORT_RESULT', success: true };
  } catch (e) {
    return { type: 'TRADE_REPORT_RESULT', success: false, error: String(e) };
  }
}
```

## Open Questions

1. **Bootstrap: How to report current state without an action_id?**
   - What we know: `POST /actions/{id}/complete` requires a valid action_id. On first run, no actions exist.
   - What's unclear: Whether the planner should add a `POST /trade-records/direct` endpoint, or redesign bootstrap to use `GET /actions/pending` first.
   - Recommendation: Add `POST /api/v1/trade-records/direct` accepting `{ea_id, action_type, price, outcome}` — inserts a TradeRecord directly, bypassing the action queue. This is the cleanest separation. Planner should include one backend task for this.

2. **Player matching: name vs ea_id vs data attribute?**
   - What we know: DOM exploration task will reveal what identifiers EA exposes on player card elements. EA's web app historically exposes player data in various ways (data attributes, aria labels, text content).
   - What's unclear: Whether ea_id is directly available as a DOM attribute or requires name-to-ea_id matching against the portfolio.
   - Recommendation: Design `readTransferList()` to return player names; the reader then resolves names to ea_ids by matching against the in-memory confirmed portfolio (stored in `portfolioItem`). If EA exposes an ID directly, use that instead.

3. **Deduplication key granularity?**
   - What we know: Keys must be unique enough to prevent double-reporting but not so fine-grained that re-listing the same player at the same price collides.
   - What's unclear: Whether `{ea_id}:{outcome}` is sufficient or needs `{ea_id}:{outcome}:{price}`.
   - Recommendation: Use `{ea_id}:{outcome}:{price}` as the key. If a player is relisted at the same price and expires again, it's a new event and should be reported.
