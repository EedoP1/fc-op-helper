## Summary

Phase 07.2 adds a Dashboard tab to the existing right sidebar overlay panel and a new backend endpoint that aggregates per-player trade state and profit statistics. The work splits cleanly into two sub-domains: (1) a new FastAPI router `GET /api/v1/portfolio/status` that joins `PortfolioSlot + TradeRecord + PlayerScore` tables, and (2) a tab bar with a Dashboard panel in the extension overlay.

The backend work closely follows the existing `profit.py` pattern ‚Äî async SQLAlchemy queries, EA tax applied in Python after SQL aggregation, and a FastAPI router with `/api/v1` prefix. The extension work wraps the existing `renderConfirmed()` section of `panel.ts` in a tab container and adds a new `renderDashboard()` function that calls a new `DASHBOARD_STATUS` message type relayed by the service worker.

The unrealized P&L computation belongs in the backend (as noted in Claude's Discretion). The backend already has `current_lowest_bin` in `MarketSnapshot` and `buy_price` in `PortfolioSlot`, so no extra API calls are needed.

**Primary recommendation:** Build backend endpoint first (it is pure Python logic with existing patterns), then wire extension tab bar and dashboard panel using the same message-relay pattern already established for portfolio operations.

## Architecture Patterns

### Backend: New `/portfolio/status` Endpoint

Location: new file `src/server/api/portfolio_status.py` (or extend `portfolio.py`)

**Recommended: separate file** ‚Äî `portfolio.py` is already ~300+ lines and the status endpoint has different join patterns. Follows the `profit.py` precedent of one concern per file.

The endpoint joins three tables:
1. `PortfolioSlot` ‚Äî all active portfolio players (buy_price, sell_price per slot)
2. `TradeRecord` ‚Äî all outcomes (bought / listed / sold / expired) per ea_id
3. `PlayerScore` ‚Äî latest scan result per ea_id for `current_lowest_bin` (unrealized P&L uses `MarketSnapshot.current_lowest_bin` via most recent `MarketSnapshot` per ea_id ‚Äî already in DB)

**Current trade status derivation (D-04):**
Derive current status from the most recent `TradeRecord.outcome` per ea_id:
- No records ‚Üí PENDING
- Last outcome = "bought" ‚Üí BOUGHT
- Last outcome = "listed" ‚Üí LISTED
- Last outcome = "sold" ‚Üí SOLD (last cycle complete, awaiting rebuy ‚Äî continuous cycle per D-05)
- Last outcome = "expired" ‚Üí EXPIRED

**Note:** `MarketSnapshot.current_lowest_bin` is what `PlayerScore` is based on, but the actual current price field lives in `MarketSnapshot` (not `PlayerScore`). Looking at `models_db.py`: `PlayerScore` does NOT have a `current_lowest_bin` column ‚Äî `MarketSnapshot` does. The CONTEXT.md (D-10) says "already tracked in PlayerScore table" but the actual model has it in `MarketSnapshot`. Use `MarketSnapshot` directly in the query, selecting the most recent snapshot per ea_id.

**Cumulative stats (D-05/D-06):**
- `times_sold`: count of `outcome == "sold"` records per ea_id
- `realized_profit`: sum sold prices √ó (1 ‚àí EA_TAX_RATE) ‚àí sum bought prices per ea_id (same logic as `profit.py`)

**Unrealized P&L per player (D-10):**
- Only meaningful for BOUGHT or LISTED status (player is held)
- `unrealized_pnl = current_lowest_bin ‚àí buy_price` for held players
- Zero/null for PENDING, SOLD, EXPIRED

**Summary bar totals (D-08):**
- `realized_profit`: sum of per-player realized profit
- `unrealized_pnl`: sum of per-player unrealized P&L (held players only)
- `trade_counts`: dict with "bought", "sold", "expired" counts across all players

### Anti-Patterns to Avoid

- **Computing unrealized P&L in the extension:** Requires the extension to know current market price, which it doesn't have without a separate fetch. Backend already has it ‚Äî keep computation server-side.
- **Combining realized + unrealized into one "total profit" figure:** D-09 explicitly prohibits this. They must be shown separately.
- **Adding `current_lowest_bin` lookup to PlayerScore model:** It's already in `MarketSnapshot`. Use that table directly.
- **Polling dashboard data automatically:** D-13 specifies manual refresh only. No setInterval in the dashboard panel.
- **Calling backend from content script directly:** Established pattern from Phase 6 ‚Äî all backend calls route through service worker.

## Common Pitfalls

### Pitfall 1: MarketSnapshot vs PlayerScore for current_lowest_bin

**What goes wrong:** CONTEXT.md D-10 says "PlayerScore table" contains `current_lowest_bin`, but `models_db.py` shows `PlayerScore` does NOT have this column ‚Äî `MarketSnapshot` does.

**Why it happens:** CONTEXT.md was written with a conceptual description rather than referencing the exact column name.

**How to avoid:** Query `MarketSnapshot` for the latest `current_lowest_bin` per ea_id. Use a subquery:
```sql
SELECT ea_id, current_lowest_bin
FROM market_snapshots
WHERE id IN (SELECT MAX(id) FROM market_snapshots GROUP BY ea_id)
```
Or a window function equivalent in SQLAlchemy.

**Warning signs:** If `PlayerScore` is queried for `current_lowest_bin`, Python will throw `AttributeError: mapped attribute 'current_lowest_bin' does not exist on PlayerScore`.

### Pitfall 2: Unrealized P&L for non-held players

**What goes wrong:** Computing unrealized P&L for PENDING or SOLD players (they don't hold coins in the card).

**Why it happens:** Summing `current_bin - buy_price` across all portfolio players without filtering by current status.

**How to avoid:** Only include unrealized P&L for players with status BOUGHT or LISTED. Set `unrealized_pnl = null` for PENDING, SOLD, EXPIRED.

**Warning signs:** Dashboard shows unrealized P&L loss for SOLD players who are waiting to rebuy.

### Pitfall 3: Current status from non-latest trade record

**What goes wrong:** Using `MAX(recorded_at)` to find the most recent trade record gives wrong results if two records have the same timestamp.

**Why it happens:** Multiple trades can be recorded in rapid succession with the same `datetime.utcnow()` value.

**How to avoid:** Use `MAX(id)` per ea_id (auto-increment integer primary key), not `MAX(recorded_at)`. The highest id is always the most recent insertion.

**Warning signs:** Status shows EXPIRED even though player was relisted immediately after.

### Pitfall 4: Tab bar visible in empty/draft states

**What goes wrong:** Tab bar renders in all panel states, violating D-02.

**Why it happens:** Tab bar added to the top of `renderConfirmed()` is fine, but if `setState('confirmed')` also touches the overall panel structure, the tab bar may leak into `renderEmpty()` or `renderDraft()`.

**How to avoid:** Tab bar creation must be entirely inside `renderConfirmed()`. The existing `container.innerHTML = ''` call at the start of each render function already clears the panel ‚Äî do not add tab bar to the outer panel structure.

### Pitfall 5: `assertNever` compile error after adding new message types

**What goes wrong:** Adding `DASHBOARD_STATUS_REQUEST` and `DASHBOARD_STATUS_RESULT` to `ExtensionMessage` union without handling them in the content script's exhaustive switch causes a TypeScript compile error on the `assertNever(msg)` default branch.

**Why it happens:** Established design (Phase 6 decision): every message type in the union must be handled in every exhaustive switch ‚Äî even if just returning `false`.

**How to avoid:** After adding types to `messages.ts`, add explicit `case 'DASHBOARD_STATUS_REQUEST': return false;` and `case 'DASHBOARD_STATUS_RESULT': return false;` to the content script switch. Same pattern as all existing `PORTFOLIO_*` and `TRADE_REPORT_RESULT` cases.

### Pitfall 6: Dashboard data fetch before tab switch

**What goes wrong:** Dashboard panel renders empty because data fetch is not triggered until after the DOM is already rendered.

**Why it happens:** Async fetch returns after synchronous render ‚Äî panel renders in "loading" state and never updates if the promise callback doesn't re-render.

**How to avoid:** Dashboard panel has two render passes: (1) render loading state synchronously, (2) after fetch resolves, re-render with data. Use a local `dashboardData` variable and a `renderDashboard()` function that can be called both at load time (shows spinner) and after data arrives (shows content). Same pattern as `renderPlayerList()` inside `renderDraft()`.

## Open Questions

1. **PortfolioSlot player names**
   - What we know: `PortfolioSlot` has no `name` column (Phase 5 decision: "player_name on TradeActions uses 'Player {ea_id}' placeholder ‚Äî PortfolioSlot stores no name"). Names are stored in `PlayerRecord`.
   - What's unclear: Are all portfolio slot ea_ids guaranteed to exist in `PlayerRecord`?
   - Recommendation: Join `PlayerRecord` for names the same way `profit.py` does ‚Äî fetch all ea_ids in one query, build a name map, fall back to `f"Player {ea_id}"`.

2. **unrealized P&L when no scan data exists**
   - What we know: Scanner may not have run yet for a recently confirmed portfolio player.
   - What's unclear: Should unrealized P&L show null, 0, or be omitted?
   - Recommendation: Return `unrealized_pnl: null` per player and `current_bin: null` when no `MarketSnapshot` row exists for that ea_id. Extension can show "‚Äî" in the UI.

3. **Dashboard data type location**
   - What we know: `storage.ts` holds extension-persisted types; `messages.ts` holds message types.
   - What's unclear: Should `DashboardData` / `DashboardPlayer` live in `storage.ts` or `messages.ts`?
   - Recommendation: Define in `messages.ts` (it is a transient message payload, not persisted to chrome.storage). This keeps `storage.ts` for items that survive worker termination only.

## Project Constraints (from CLAUDE.md)

- **Data source:** fut.gg API only. No new data sources introduced in this phase ‚Äî all data comes from existing DB tables populated by the scanner.
- **Tech stack:** Python backend + TypeScript Chrome extension. No new languages.
