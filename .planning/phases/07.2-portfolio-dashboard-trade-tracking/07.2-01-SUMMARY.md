# Phase 07.2 Plan 01: Portfolio Status Endpoint Summary

**One-liner:** Backend GET /api/v1/portfolio/status with per-player trade status, unrealized P&L, and summary totals using SQLAlchemy subquery joins.

## What Was Built

A new FastAPI endpoint that serves as the single data contract for the extension dashboard (D-07). The endpoint queries four tables in one async session and assembles a complete portfolio snapshot per player.

### Endpoint: GET /api/v1/portfolio/status

Response shape:
```json
{
  "summary": {
    "realized_profit": 16500,
    "unrealized_pnl": 5000,
    "trade_counts": {"bought": 2, "sold": 1, "expired": 0}
  },
  "players": [
    {
      "ea_id": 1001,
      "name": "Kylian Mbappe",
      "status": "SOLD",
      "times_sold": 1,
      "realized_profit": 16500,
      "unrealized_pnl": null,
      "buy_price": 50000,
      "sell_price": 70000,
      "current_bin": null
    }
  ]
}
```

**Status states:** PENDING (no trade records), BOUGHT, LISTED, SOLD, EXPIRED — derived from latest TradeRecord.outcome.

**Unrealized P&L:** Only computed for BOUGHT/LISTED players with a MarketSnapshot. PENDING/SOLD/EXPIRED get null.

**Realized profit formula:** `int(total_sold_gross * (1 - EA_TAX_RATE)) - times_sold * buy_price` — uses slot.buy_price as cost basis per completed cycle, not total trade record buy costs.

## Decisions Made

1. **Realized profit cost basis**: Used `times_sold * slot.buy_price` instead of `total_bought_cost` from trade records. This ensures open positions (BOUGHT/LISTED) show 0 realized profit rather than negative values. The plan test assertions confirmed this interpretation.

2. **MAX(id) for latest records**: Both latest TradeRecord and latest MarketSnapshot use `func.max(id)` subqueries per the plan's explicit pitfall guidance — avoids issues with same-timestamp records.

3. **current_bin returned as None for non-held statuses**: Even if a MarketSnapshot exists for a SOLD player, current_bin is returned as null since the market price is irrelevant to closed positions.
