# Phase 07.2: Portfolio Dashboard & Trade Tracking - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a Dashboard tab to the existing extension overlay panel that shows per-player trade status (current cycle position + cumulative stats), realized profit, unrealized P&L based on current market prices, and trade counts. Backend provides a dedicated status endpoint. No automation (Phase 8), no activity feed (future enhancement). Depends on Phase 07.1 (Trade Reporting) for data.

</domain>

<decisions>
## Implementation Decisions

### Dashboard Layout
- **D-01:** Add a tab bar to the existing right sidebar overlay. Two tabs: "Portfolio" (existing view) and "Dashboard" (new trade tracking view).
- **D-02:** Tab bar only visible in the confirmed overlay state. Empty and draft states remain as-is — no dashboard to show before portfolio is confirmed.
- **D-03:** Portfolio tab is always the default landing tab. User switches to Dashboard manually.

### Trade Activity Display
- **D-04:** Dashboard shows per-player trade status list. Each player shows their current position in the trade cycle: PENDING (not yet bought), BOUGHT (awaiting list), LISTED (on market), SOLD (cycle complete), EXPIRED (listing expired, awaiting relist).
- **D-05:** Trade cycle is continuous — after a player is sold, they re-enter the buy cycle. Dashboard tracks cumulative stats: total times sold and total profit earned per player across all cycles.
- **D-06:** Per-player display: player name, current status badge, plus "Nx sold • +X profit" cumulative line underneath. Compact two-line format per player.
- **D-07:** Backend provides a new dedicated `GET /api/v1/portfolio/status` endpoint that returns each player's current trade state plus cumulative stats. Single call, clean contract — extension does not derive status from multiple endpoints.

### Profit Visibility
- **D-08:** Dashboard summary bar at top shows: realized profit (from completed sell cycles), unrealized P&L (current market value vs buy price for held players), and trade counts (bought/sold/expired).
- **D-09:** Realized and unrealized profit displayed as separate lines, side by side. NOT combined into a single total. Each color-coded independently (green positive, red negative).
- **D-10:** Unrealized P&L uses current lowest BIN from the scanner's price data (already tracked in PlayerScore table). No extra API calls needed — data already in DB.
- **D-11:** Per-player profit numbers are also color-coded (green/red).

### Data Freshness
- **D-12:** Dashboard fetches fresh data on tab switch (switching to Dashboard tab triggers a fetch).
- **D-13:** Manual refresh button for subsequent updates while on the Dashboard tab. No auto-polling.

### Claude's Discretion
- Exact styling of tab bar, status badges, and color values for green/red on dark theme
- Loading/spinner state while dashboard data is being fetched
- Sort order of players in the dashboard list (by status, by profit, by name)
- Error handling for failed dashboard data fetch
- Response shape of the new `/portfolio/status` endpoint (as long as it provides the data needed for D-04 through D-10)
- Whether unrealized P&L is computed in backend or extension (backend recommended since it has price data)

</decisions>

<specifics>
## Specific Ideas

- User specifically wants to track how many times each player has been sold and total profit per player across all cycles (not just current cycle)
- Realized and unrealized profit must be visually separate — never combined into one number
- Trade cycle is continuous: buy → list → sell → buy again. Dashboard must reflect this loop, not treat "sold" as terminal state
- Unrealized P&L based on current lowest BIN from scanner data (already in PlayerScore table)

</specifics>
