DELETE /api/v1/portfolio/{ea_id}?budget=N — remove player and get replacements
  Response 200: {"removed_ea_id": int, "freed_budget": int, "replacements": [...]}

  Request: {"ea_id": int, "price": int, "outcome": str}

  Request: {"records": [{"ea_id": int, "price": int, "outcome": str}]}

GET /api/v1/portfolio/status — portfolio status with per-player trade info
  Response 200: {"summary": {...}, "players": [{...}]}

GET /api/v1/profit/summary — profit aggregation
  Response 200: {"totals": {"total_spent": int, "total_earned": int, "net_profit": int, "trade_count": int}, "per_player": [...]}

Lifecycle state machine (from _derive_next_action):
  No records     -> BUY
  bought         -> LIST
  listed         -> waiting (skip)
  sold           -> BUY (restart)
  expired        -> RELIST

1. `test_full_buy_list_sold_cycle` (D-07, D-10):
   - Seed 1 slot (POST /portfolio/slots with real ea_id, buy_price, sell_price from generate response)
   - GET /actions/pending -> assert action_type == "BUY", extract action_id
   - POST /actions/{id}/complete with outcome="bought", price=buy_price
   - GET /actions/pending -> assert action_type == "LIST", extract action_id
   - POST /actions/{id}/complete with outcome="sold", price=sell_price
   - GET /actions/pending -> assert action_type == "BUY" (new cycle starts per lifecycle state machine)
   - GET /portfolio/status -> assert player status == "SOLD" (most recent record)

   - GET /profit/summary -> assert totals.net_profit != 0 (there was a sale)

2. `test_buy_list_expired_relist_cycle` (D-10):
   - Seed 1 slot
   - BUY -> complete(bought) -> LIST -> complete(expired) ->
   - GET /actions/pending -> assert action_type == "RELIST"

3. `test_listed_means_waiting` (D-10):
   - Seed 1 slot
   - BUY -> complete(bought) -> LIST -> complete(listed)

Create tests/integration/test_concurrent_removes.py targeting the known concurrent-remove duplicate bug and other race conditions. These tests use asyncio.gather to fire multiple requests simultaneously.

**Tests to write:**

1. `test_concurrent_remove_two_players_no_duplicates` (D-08 — the known bug):
   - POST /portfolio/generate with budget=200000, get player list
   - POST /portfolio/confirm with the generated players
   - Pick 2 players from the confirmed list
   - Fire 2 concurrent DELETE /portfolio/{ea_id}?budget=200000 requests using asyncio.gather
   - Both should return 200
   - Collect all replacement ea_ids from both responses
   - GET /portfolio/confirmed — get current portfolio
   - Assert NO duplicate ea_ids in the portfolio (this is where the bug manifests)
   - Assert the 2 removed players are NOT in the portfolio
   - Assert total portfolio count is reasonable (original count - 2 + replacements)

2. `test_concurrent_remove_three_players` (D-08 extended):
   - Same as above but with 3 concurrent removes
   - More likely to trigger race conditions

3. `test_rapid_pending_action_polling` (D-11):
   - Seed 1 slot
   - Fire 10 concurrent GET /actions/pending using asyncio.gather
   - All should return 200
   - At most 1 should have a non-null action (action should be claimed by exactly one)
   - Actually, the real behavior: the first call derives and claims the action. Subsequent concurrent calls should either see the IN_PROGRESS action or return null. Due to SQLite serialization, they may all see the same IN_PROGRESS action (idempotent return). Assert: all responses are valid (200), all non-null actions have the same id (no duplicate actions created).

4. `test_rapid_complete_same_action` (D-11):
   - Seed 1 slot, get pending action
   - Fire 3 concurrent POST /actions/{id}/complete with outcome="bought"
   - First should succeed (200), subsequent may return 200 too (since complete doesn't guard against re-completing per Phase 09 decision)
   - Key assertion: GET /actions/pending after should derive LIST (not corrupt state)
   - GET /portfolio/status should show BOUGHT status (not inconsistent)

5. `test_concurrent_slot_seeding` (D-11):
   - Fire 5 concurrent POST /portfolio/slots with the same ea_id
   - Should not create duplicate PortfolioSlot rows (ea_id is UNIQUE)
   - GET /portfolio/confirmed should show at most 1 entry for that ea_id

6. `test_concurrent_direct_trade_records` (D-11):
   - Seed 1 slot
   - Fire 5 concurrent POST /trade-records/direct with same ea_id and outcome="bought"
   - Server-side dedup should prevent duplicate records
   - Only 1 trade record should be created (subsequent calls deduped)
   - GET /actions/pending should correctly derive LIST

7. `test_remove_during_action_lifecycle` (D-10):
   - Seed 2 slots
   - GET /actions/pending -> get BUY action for slot 1
   - While action is IN_PROGRESS, DELETE /portfolio/{ea_id} for slot 1
   - The delete should cancel the pending/in-progress action
   - GET /actions/pending should now return action for slot 2 (not the cancelled one)

IMPORTANT per D-06: Concurrent access patterns are critical. The real extension sends rapid API calls. Use `asyncio.gather` with real httpx.AsyncClient connections to the real server.

Per D-04: Tests that fail = server bugs. The concurrent remove duplicate bug is a KNOWN issue. If test_concurrent_remove_two_players_no_duplicates fails, that confirms the bug exists and needs fixing.
